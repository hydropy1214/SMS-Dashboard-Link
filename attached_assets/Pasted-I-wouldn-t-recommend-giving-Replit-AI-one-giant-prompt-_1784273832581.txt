I wouldn't recommend giving Replit AI one giant prompt to "change everything." For a project of this size, that usually leads to broken code, inconsistent architecture, and difficult debugging.

A better approach is to have it work from a detailed Product Requirements Document (PRD) and implement the changes in phases. Here's a comprehensive PRD you can give it as the starting point.

# Dispatch v2 - Complete Architecture Refactor PRD

## Goal

Transform the current Dispatch project into a production-grade self-hosted messaging management platform.

The system must remain self-hosted. The Mac Agent is responsible for controlling Messages.app. The web dashboard manages users, campaigns, devices, monitoring, and job scheduling.

The project should prioritize modularity, maintainability, security, observability, and future scalability.

---

# Overall Architecture

```
Dashboard (React)

↓

Express API

↓

PostgreSQL

↓

Redis Queue

↓

Mac Agent Manager

↓

Secure HTTPS

↓

Mac Agent

↓

Messages.app

↓

iMessage / SMS
```

The Mac Agent is the only component allowed to interact with AppleScript.

No browser or frontend component may communicate directly with the Mac Agent.

---

# Major Refactor Requirements

Completely reorganize the project into modular services.

Remove any business logic from React components.

All business logic belongs in the API server and Mac Agent.

---

# Future Scalability

Design the system so one account can manage:

Many Macs

Many campaigns

Many users

Many sender identities

Many queues

without changing the architecture.

The codebase should be modular, production-ready, and easy to extend while preserving the existing self-hosted model where the Mac Agent controls Messages.app.

After this refactor is complete and stable, a second phase can add features such as advanced scheduling, templates, contact management, user roles, notifications, and richer analytics. Trying to implement everything in one pass is likely to make the project unstable, so it's better to complete and verify the architectural refactor before expanding functionality.
